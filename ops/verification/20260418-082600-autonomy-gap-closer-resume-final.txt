--- root cause identified ---
`ops/task_runner_preflight.py::run_preflight()` previously wrote a
timestamped `ops/verification/YYYYMMDD-HHMMSS-preflight.txt` on *every*
invocation — including clean no-op ticks. The runner's commit/push loop
then added those files to git, which updated `refs/heads/main` +
`FETCH_HEAD`, which re-fired the launchd WatchPath, which re-invoked the
runner, which re-ran the preflight, which wrote another report. This
positive-feedback loop accumulated to:

* 3,082 preflight reports under `ops/verification/` (tracked, committed).
* 177 local heartbeat commits diverging from origin/main after the remote
  moved (`05d960d`, `06bcab7` watchdog commits) — `git push origin main`
  was silently non-fast-forward and never reached the remote.

--- minimal fix applied ---
File: ops/task_runner_preflight.py

1. New helper `_preflight_did_work(result)` returns True only when
   preflight actually mutated state: gitattributes updates, whitelisted
   conflicts resolved, whitelisted dirty staged, unsafe conflicts,
   notes present, or result.ok == False.

2. `run_preflight()` now calls `write_report(result)` only when
   `_preflight_did_work()` is True. Otherwise it stamps `finished_at`
   and returns without touching disk. Clean ticks produce no artefact,
   so no launchd re-trigger is possible.

3. No changes to the write-report path itself — when preflight really
   does heal a conflict, a single timestamped report still lands in
   ops/verification/ as before, and the commit_and_push loop continues
   to push it.

4. No changes to the runner, gates, queue-status, audit-index, or
   approvals tools. Those were correct as-is.

--- dry-run path documentation ---
Both high-risk and low-risk tasks can opt into dry-run via either:
  * top-level `"dry_run": true` on the task JSON, or
  * `payload.dry_run = true`.
The gate module treats dry_run as a universal high-risk bypass (no side
effects). scripts/task_runner.py propagates the flag into the payload
shallow-copy so handlers that understand `--dry-run` (run_cline_prompt,
run_cline_campaign) forward it to their launchers. Handlers that cannot
honor dry-run should never be flagged high-risk — the gate module
recognises only dry_run as an authority bypass.
