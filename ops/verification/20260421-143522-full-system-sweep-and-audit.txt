scripts/task_runner.py:608 routes verify_dump tasks to handle_verify_dump,
which tees a run into a new ops/verification/<stamp>-<name>.txt file. After
the handler returns rc=0 the pending file should move to completed/. It
doesn't — the verify script is writing ~stamp~-watchdog-install.txt
successfully every tick AND the task json stays in pending/ (visible: same
file name is the only occupant of ops/work_queue/pending/).

Root cause (two nested issues):
  (a) ops/task_runner/verifications/watchdog-install.sh (or equivalent)
      sources /Users/bob/AI-Server/.env. Line 348 is:
          ACH_BANK_NAME=First Bank of Colorado
      Under `set -euo pipefail`, `source .env` interprets "Bank" as a
      command after ACH_BANK_NAME=First. Error in every result file:
          /Users/bob/AI-Server/.env: line 348: Bank: command not found
      The step keeps going (heartbeat etc. is found) but the overall
      sourcing is non-deterministic, and the task never completes cleanly
      from the runner's POV. Fix: quote the value, or switch to
      `scripts/api-post.sh`-style env loading via
      `export $(grep -vE '^(#|$)' .env | xargs)` in the verify script.
  (b) Even if (a) were benign, the runner thinks the task is pending
      because the pending JSON is never moved. Either the handler is
      returning non-zero on a subtle step or ops/work_queue/pending/ is
      not being scanned as completed after this specific verify_dump.
      Needs a targeted scripts/task_runner.py trace run with DEBUG=1 to
      confirm which branch hits. Short-term: manually `git mv
      ops/work_queue/pending/20260417-170500-install-watchdog.json
      ops/work_queue/completed/` and push.

Blast radius: we've consumed ~140 KB/h of disk + git churn since 2026-04-17
on files that carry no new information. It also drowns out legit artifacts
in ops/verification/. The verification-index.sh and verification-prune.sh
helpers can clean it, but the source needs to stop first.
