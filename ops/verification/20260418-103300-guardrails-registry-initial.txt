--- tie-back to lessons + evidence ---
These bootstrap guardrails are policy rules that already exist elsewhere in
the repo (`CLAUDE.md`, `ops/AGENT_VERIFICATION_PROTOCOL.md`,
`.clinerules`). The `derived_from_lessons = manual` column records that.
Any future guardrail added via the learning loop will point at one or more
`L-*` ids in `ops/LESSONS_REGISTRY.md`; the miner ensures those rows carry
`evidence_refs` back to specific `ops/verification/<stamp>-*.txt` files.

--- how future agents add a guardrail ---
1. Open ops/LESSONS_REGISTRY.md. Find the relevant lesson_id (L-XXXXXXXX)
   and flip its status to `promoted_to_guardrail`.
2. Open ops/GUARDRAILS.md. Append a row under "Active guardrails" with
   the next G-NN id, `derived_from_lessons = <lesson_id>` (pipe-separated
   if multiple), and filled-in scope / risk_tier / enforcement / verification.
3. If the guardrail introduces automated enforcement, update the relevant
   script (preflight, gate, launchd plist, prompts).
4. Commit both files in the same change. Do not delete old rows — set
   `status = deprecated` for rules that are superseded.

--- limitations / next improvements ---
- No automated "promote this lesson" CLI yet. The promotion step is a
  structured manual edit — intentional for this first pass, so humans stay
  in the loop on what becomes a rule.
- `ops/learning_digest.py` surfaces the active guardrails table each run,
  which gives a cheap cross-check that the file is parseable and that no
  row has been accidentally dropped.
- G-07 is advisory today; the miner currently only populates
  `evidence_refs` from verification file names, not from Decision Journal
  / Follow-ups DB rows. Adding that will happen when the miner gets its
  next upgrade.
