=== Module Structure (cortex/autonomy.py) ===
Data models    : Verification, HumanGate, AutonomyQuestion, AutonomyOverview (dataclasses)
VerificationScanner
  - Scans ops/verification/, up to 50 most-recent files
  - Bounded: first 40 lines, skip >500 KB files
  - Parses: path, timestamp, topic, verdict (PASS/FAIL/PARTIAL/GAP/UNKNOWN/ARMED/CLOSED/NEEDS_MATT), summary
HumanGateScanner
  - Scans STATUS_REPORT.md (last 200 lines)
  - Scans ops/runbooks/ and .cursor/prompts/ filenames for gate keywords
  - Markers: NEEDS_MATT, [FOLLOWUP], ARMED, blocked, waiting, confirmation required
  - Returns up to 20 gates, most-recent first
AutonomyAssessor
  - Answers 10 autonomy questions via cheap local checks (httpx, fs, env)
  - Returns AutonomyOverview with overall_status (ok/warn/degraded)
register_autonomy_routes(app)
  - Mounts GET /api/autonomy/overview on FastAPI app

=== Known Limitations ===
- HTTP checks use localhost; when cortex is not running (e.g. during testing),
  is_bob_alive / can_receive_messages / can_send_outbound will show "fail/warn" —
  this is expected behavior (they should degrade gracefully).
- BlueBubbles health check falls back to "warn" rather than "fail" when the
  endpoint is unreachable, because BlueBubbles absence does not prevent all
  autonomy. NEEDS_MATT gate marker detection in STATUS_REPORT.md scans only
  the last 200 lines — earlier history is not surfaced (intentional).
- No persistent caching; each /api/autonomy/overview call re-scans disk.
  For a high-traffic dashboard this is acceptable (files are small, reads fast).
- Embeddings check is env-only; no live Ollama ping in this version.
