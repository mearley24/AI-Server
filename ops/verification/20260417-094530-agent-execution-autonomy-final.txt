------------------------------------------------------------------------------
Pros
------------------------------------------------------------------------------
- Zero-touch: no human in the loop for any low-risk operation.
- ed25519 + allowlist + named-script discipline keeps a git-driven
  runner safe.
- Event-driven plist delivers sub-second dispatch with a 60s safety
  net and 10s throttle.
- 3-tier safety: lock + timeout + heartbeat.
- Handlers are small + auditable; each action logs to
  ops/verification/<task_id>-result.txt.
- 13 tasks succeeded today end-to-end — proves the plane works.

------------------------------------------------------------------------------
Cons / weaknesses
------------------------------------------------------------------------------
- No first-class chaining (`depends_on`) metadata.
- No priority; pending sorted by filename.
- No automatic retry for transient failures.
- Failures invisible without polling failed/ or reading verification.
- Runtime .runner.lock was tracked in git — FIXED this pass.
- Allowlist hard-coded in Python, not a config file.
- No batch signing helper, no dry-run flag on the runner.
- No key-rotation audit trail.

------------------------------------------------------------------------------
Better replacements or package suggestions
------------------------------------------------------------------------------
- Celery/RQ/Dramatiq/Temporal/Airflow: all rejected. Broker/worker
  failure modes worse than plain git+launchd. Skip.
- GitHub Actions: external; violates "everything stays on Bob".
- Current design is the right shape. Improvements = additive.
