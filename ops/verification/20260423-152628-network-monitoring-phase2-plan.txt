2. NEXT-STEP RATIONALE — what should Phase-2 do, and in what order?
===============================================================================

The two blockers are linked: arming the dropout-watch LaunchAgent is safe in
isolation, but it only gives us a *second* monitoring channel.  The
network-guard crash is the root cause of the Apr-3-onward health-record gap,
and fixing it requires a new repo-owned file (tools/security_utils.py) plus a
small sys.path bootstrap in two callers.

Therefore Phase-2 should:

  Step 1.  Create tools/security_utils.py as a minimal, stdlib-only shim
           providing the five symbols the two callers import:
             - sanitize_for_telegram
             - hash_text, mask_contact, mask_name, redact_text

  Step 2.  Insert a 3-line sys.path bootstrap at the top of both
           tools/network_guard_daemon.py and tools/imessage_watcher.py so the
           LaunchAgent-invoked script finds security_utils without adding
           PYTHONPATH to the plist (smaller blast radius than env munging).

  Step 3.  Dry-run the network guard in --once mode with Telegram env vars
           unset to prove the import works without triggering an external
           send.

  Step 4.  launchctl bootstrap the dropout-watch LaunchAgent; verify
           data/network_watch/dropout_watch_status.json shows
           running=true and health=healthy.

  Step 5.  launchctl bootout + bootstrap the existing network-guard
           LaunchAgent so it picks up the fixed import; wait one 60 s tick
           and verify a fresh log line + data/network_guard_state.json mtime.

  Step 6.  Truncate logs/network-guard.err (8 MB of stale tracebacks).

  Step 7.  Commit the repo changes (security_utils.py + two caller patches +
           verification artifact + audit doc + STATUS_REPORT update) and
           push to origin/main.

All of Phase-2 happens on Bob (Cline-first) because this sandbox cannot
launchctl bootstrap anything.  The durable repo artifact that drives it is
the new prompt at:
