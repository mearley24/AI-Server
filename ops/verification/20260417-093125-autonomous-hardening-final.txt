2. AUTONOMOUS BEHAVIORS ADDED
---------------------------------------------------------------------

A. Standing approval model (CLAUDE.md)
   - Low-risk work (diagnostics, verification, repo hygiene, health
     checks, queue inspection, logging, prompt updates, tooling,
     internal automation) is pre-authorized — agents just do it and
     log the result to ops/verification/.
   - Medium-risk work (service restarts, non-secret env changes,
     launchd installs, non-financial migrations) proceeds without
     synchronous approval but MUST leave a verification trail.
   - High-risk work (data deletion, secrets rotation, money moves,
     trading, customer-visible comms, destructive infra) still
     requires explicit approval before execution.

B. Preflight self-heal (ops/task_runner_preflight.py)
   - Runs before every task-runner tick (wired into scripts/task_runner.py).
   - Heals whitelisted state-file conflicts automatically (currently
     knowledge/markup_exports/.session_tracking.json and
     data/cortex/digests/**).
   - Ensures .gitattributes merge=ours rules are present; appends
     missing ones and commits the change.
   - Commits + pushes its own safe changes; never touches user code.
   - Writes a timestamped ops/verification/<stamp>-preflight.txt
     report every time it runs meaningful work.
   - Non-whitelisted conflicts are reported in detail and cause the
     runner to SKIP task dispatch (but the heartbeat still ticks).

C. Auto-follow-up verification
   - task_runner_preflight.py writes its own report.
   - task_runner_health.py writes its own report.
   - task_audit.py --out persists an audit report.
   - Every task the runner executes already writes
     ops/verification/<task_id>-result.txt.

D. Health + audit tooling
   - task_runner_health.py: five named checks (launchd, heartbeat,
     queue, verification, preflight). Exit 0 healthy / 1 unhealthy.
   - task_audit.py: one-shot forensics for any task id or substring.

E. Interactive-prompt hazard pre-emption
   - Agent Verification Protocol now explicitly states: ssh_and_run
     tasks must pin host keys before running; no TTY editors; bounded
     scripts only.

F. Blocker-report pattern
   - Agents write ops/verification/<stamp>-blocker-<topic>.txt instead
     of asking Matt to paste anything. Four required fields documented.

      launchctl load ~/Library/LaunchAgents/com.symphony.task-runner-health.plist
  Loading can be done via a signed run_script task or by a human.

- preflight.run_preflight() commits + pushes on every invocation when
  it finds whitelisted dirty state — this is the intended behavior but
  it produces one extra commit per tick while state files change.
  Acceptable for now; can be throttled later if commit noise becomes
  an issue.

- task_audit.py is a local-read tool; it does not follow the queue
  across ssh_and_run remote hosts. Remote task trails are captured via
  the existing ssh_and_run tee-to-file pattern.

- task_runner_preflight.py uses `git checkout --ours` which requires
  the file to be a real conflict entry or locally modified. If it ever
  needs to accept-ours on a deleted-by-them conflict, an explicit case
  would need to be added.
