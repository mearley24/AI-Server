3. Pros
------------------------------------------------------------------------------
- Zero-touch control plane. Matt is not in the loop. Every operation
  produces an auditable verification artifact.
- ed25519 signing + allowlist makes the runner safe even though tasks
  arrive by plain git push.
- Event-driven plist (WatchPaths + QueueDirectories) gives sub-second
  response to new tasks while StartInterval=60 is a safety net.
- ThrottleInterval=10s prevents thundering-herd on multi-file commits.
- Preflight self-heal + whitelist keeps the runner unblocked on
  well-understood runtime state files.
- Handler set is small (4) and each handler is tightly scoped — easy
  to reason about.
- Heartbeat + per-task timeout + single-instance lock form three
  independent safety nets.
- Named-script discipline (run_script / verify_dump / ssh_and_run) makes
  every remote/local action reviewable in git history.

------------------------------------------------------------------------------
4. Cons / weaknesses / missing pieces
------------------------------------------------------------------------------
- No task chaining metadata. If step N fails, steps N+1..M stay in
  pending until the submitter removes them. Workflow orchestration
  lives entirely in the submitter's head.
- `failed/` and `rejected/` have no alerting. A task that dies silently
  is only visible when a human reads the dir.
- No task replay / retry semantics. A transient network blip that
  causes a failed ssh_and_run requires the submitter to manually
  re-queue a near-identical task.
- data/task_runner/.runner.lock is tracked in git (appears in
  `git status -s` as modified). It should be in .gitignore — the
  lock file is pure runtime state. Every runner tick creates a churn
  opportunity.
- No task "owner" field. Every task carries `created_by` but the
  runner doesn't route results back to any channel — the submitter
  must poll the repo.
- task_signer.py has no signature revocation mechanism. If a private
  key leaks, the only recovery is to delete the pubkey line from
  AUTHORIZED_KEYS.txt (which works, but there's no audit of who had
  access to a key).
- No batch signing for agents who want to queue many tasks quickly.
- No `dry-run` mode on the runner itself for smoke testing a new
  handler without side effects.
- Allowlist is hard-coded in task_runner.py. Adding a new task type
  requires editing Python, not a config file.
- No task priority. Runner processes pending in sorted filename order
  (effectively timestamp-ordered), which is fine, but any emergency
  task must have a filename that sorts before pending items.

------------------------------------------------------------------------------
5. Better replacements or package suggestions
------------------------------------------------------------------------------
- Celery / RQ / Dramatiq: full-fat task queues. Overkill. Their
  failure modes (broker down, worker crash) are worse than the current
  git-driven pipeline. Don't adopt.
- Temporal / Airflow: workflow engines. Massively overkill and would
  be harder for humans to inspect than "read a JSON file in a repo".
- GitHub Actions / Workflow Dispatch: external, would break the
  tenet of "everything stays on Bob".
- Hashicorp Vault + OIDC: proper key rotation. If key leakage becomes
  a real concern, consider — but for two agents and a closed repo,
  AUTHORIZED_KEYS.txt with git history is sufficient.

Net: current system is the right shape. Improvements are additive
(alerting, chaining, gitignore fixes), not replacements.
