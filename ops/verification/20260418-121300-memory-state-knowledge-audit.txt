6. Better replacements (considered, not recommended yet)
------------------------------------------------------------------------------
PostgreSQL + pgvector
  - Pros: strong vector support, ACID, better concurrency, server
    model.
  - Cons: entire stack rewrite, cross-container backups harder,
    violates the "bind-mounted-SQLite is simple" posture that
    makes this whole system inspectable with sqlite3 CLI.
  - Verdict: not today. Revisit when brain.db crosses 500 k rows
    or when concurrent writers become a problem.

LiteFS (streaming replication layer over SQLite)
  - Pros: minimal disruption; keeps sqlite3 CLI; gives remote
    replica for backup and disaster recovery.
  - Cons: adds a process + FUSE mount; overkill for single-node
    use case.
  - Verdict: consider alongside litestream as a read-replica
    story if a second node is added.

Chroma / LanceDB / sqlite-vss
  - Pros: local, embeddable, keeps the single-host posture.
    sqlite-vss specifically can be loaded as an extension onto
    brain.db without rearchitecting the store.
  - Cons: requires running an embedder (OpenAI or local) per
    insert. Today's trading_strategy write rate (12 k/day) would
    cost ~$0.12/day in embeddings via text-embedding-3-small.
  - Verdict: plan for it; see §8.

Structured "mistakes registry" layered on top of Cortex
  - Pros: the scaffolding already exists — cortex.db has
    `recurring_problems` + `problem_occurrences` tables. A thin
    ingestion path from ops/verification/*-blocker-*.txt into a
    canonical "known_mistake" category in brain.db (or reviving
    cortex.db) would close the loop.
  - Cons: requires convention discipline. Mistake rows need
    stable keys ("blocked_on_missing_secret:KRAKEN_SECRET") so
    repeats are detectable.
  - Verdict: this is the highest-leverage upgrade, strictly
    additive over Cortex. Deferred to its own prompt.

Net: Cortex + SQLite remains the right stack today. The wins are
enforcement hygiene (§5.1-§5.3) and per-category TTL, not a
datastore swap.
