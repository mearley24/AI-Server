3. Pros
------------------------------------------------------------------------------
- Extremely strong verification culture. Every tick writes a timestamped
  artifact; 48 files since ~09:00 this morning.
- Preflight + whitelist self-heal prevents pull conflicts without human
  intervention.
- Task-runner health check is a proper machine-readable JSON+human txt
  artifact with exit codes — wireable into launchd alerting.
- bob-watchdog.sh is multi-layer, self-bounded, cooldown-protected.
- .gitattributes + pull.sh combine to make `bash scripts/pull.sh` the
  safe, idempotent git entrypoint.
- Multiple redundant self-healing layers (system, Docker, repo, service).

------------------------------------------------------------------------------
4. Cons / weaknesses / missing pieces
------------------------------------------------------------------------------
- scripts/verify-deploy.sh still probes mission-control 8098 (removed).
  Every run emits a FAIL for a service that no longer exists. This is
  actively misleading any agent reading the artifact.
- scripts/verify-deploy.sh does not probe cortex (8102), email-monitor,
  notification-hub, proposals, calendar-agent, polymarket-bot, x-intake.
  Its coverage is narrower than reality.
- No single launchd inventory doc: 47 plists, no map. When one silently
  fails its target script is missing, there is no sweeper to catch it.
- No `ops/verification/` retention policy — grows unbounded. With ~50
  artifacts/day it will reach 5–10k within ~6 months. Eventually `ls`
  gets slow and dashboard widgets that enumerate recent artifacts bog.
- No aggregated "last N verification artifacts" summary — agents and
  dashboards must list the dir and sort. A rotating `ops/verification/
  INDEX.txt` that names the newest per topic would be cheap.
- Heartbeat staleness threshold (900s) is generous. A silent runner
  failure won't alarm for 15 min. Acceptable for v1 but no alerting
  route when `ok=False` beyond exit code.
- No `scripts/doctor.sh` style aggregator. An operator must run
  verify-deploy, smoke-test, task_runner_health, and bob-watchdog status
  separately. Each produces a different artifact format.
- CLAUDE.md lesson #17 (sell haircut rounding) still UNKNOWN.
- Dropbox link validator (lesson #4) still absent.

5. Better replacements or package suggestions
------------------------------------------------------------------------------
- Supervisor-style health: replacing stdout/JSON aggregates with a
  lightweight Prometheus `node_exporter` + textfile-collector would give
  a time series. Not worth the complexity yet — Redis events:log
  already covers recency. Skip.
- structlog (pypi) for Python services — unified JSON structured log
  output. Would reduce the ad-hoc logger.info lines. Low priority.
- healthchecks.io external ping for bob-watchdog + task-runner. Adds a
  remote dead-man switch without moving infra. Optional.
- None of the above worth the churn today. Current stack is
  sufficient — fix the stale probes and add a retention+index layer
  instead.
